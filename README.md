# Epistola

> **Современный корпоративный мессенджер на Flutter и Firebase**

Epistola — долгосрочный проект корпоративной коммуникационной платформы. Сейчас это Android-first мессенджер на Flutter и Firebase, но архитектура постепенно готовится к Web/iOS, модульным рабочим пространствам Spaces, медиа-подсистеме, возможной смене backend-хранилища и будущему собственному backend.

## ⚡ Что такое Epistola

**Epistola** — корпоративный мессенджер, который создаётся как единая коммуникационная платформа для сотрудников компании.

Краткосрочная цель — стабильный мессенджер для пилотной группы примерно 40–50 пользователей.

Долгосрочная цель — рабочая платформа для компании на 600–700 сотрудников с чатами, группами, ролями, Spaces, мини-приложениями, файлами, рабочими сменами, задачами, объявлениями, документами и внутренними сервисами.

Проект развивается по принципу:

> Сначала надёжная архитектура, затем функциональность поверх неё.

## 🧭 Архитектурная идея проекта

Epistola больше не рассматривается как просто Flutter-приложение.

Текущая архитектурная цель:

```text
Flutter client
    ↓
Domain models
    ↓
Services / Facades
    ↓
Providers / Infrastructure adapters
    ↓
Firebase сейчас
    ↓
Собственный backend / S3 / другое хранилище в будущем
```

Ключевые принципы:

- UI не должен знать детали инфраструктуры.
- Firebase используется сейчас, но не считается вечной зависимостью.
- Domain-модели не должны зависеть от Flutter, Firebase, Storage или UI.
- Внешние сервисы должны быть обёрнуты в сервисы/провайдеры.
- Новые крупные подсистемы сначала проектируются, затем реализуются.
- Код должен проходить `flutter analyze` без ошибок.

Media Foundation — это новая подсистема, на которой позже будут построены:

- видео;
- локальный кэш;

### Слои Media Foundation

```text
UI
 │
 ▼
AvatarService / AttachmentService (future)
 │
 ▼
MediaStorageService
 │
 ▼
MediaStorageProvider
 │
 ├── FirebaseMediaStorageProvider
 ├── S3MediaStorageProvider (future)
 ├── YandexStorageProvider (future)
 ├── GoogleDriveStorageProvider (future)
 └── EpistolaBackendStorageProvider (future)
```

### Что делает каждый слой

| Слой | Ответственность |
| --- | --- |
| `MediaAsset` | Описание медиа-объекта как доменной модели |
| `MediaPaths` | Единое место генерации путей хранения |
| `MediaStorageProvider` | Абстрактный контракт хранилища |
| `FirebaseMediaStorageProvider` | Реализация хранения через Firebase Storage |
| `MediaStorageService` | Фасад для приложения, скрывающий конкретный provider |

### Почему это важно

Приложение не должно быть жёстко связано с Firebase Storage.

Если в будущем Epistola перейдёт на:

- европейский Firebase/GCS bucket;
- S3-compatible storage;
- Yandex Object Storage;
- Cloudflare R2;
- Google Drive;
- собственный backend;
- локальный сервер компании;

то основная бизнес-логика и UI не должны переписываться. Достаточно будет заменить или добавить реализацию `MediaStorageProvider`.

## 🗂 MediaAsset

`MediaAsset` — это не сам файл, а паспорт файла.

Он описывает:

- уникальный id;
- provider;
- путь хранения;
- тип медиа;
- владельца;
- MIME type;
- размер;
- ширину и высоту;
- длительность;
- версию;
- даты создания/обновления;
- опциональный download URL.

Примерная логика:

```text
MediaAsset
 ├── id
 ├── provider
 ├── path
 ├── type
 ├── ownerType
 ├── ownerId
 ├── mimeType
 ├── sizeBytes
 ├── width / height
 ├── durationMs
 ├── version
 ├── createdAt / updatedAt
 └── downloadUrl?
```

`downloadUrl` не является главным источником истины. Основные поля — `provider` и `path`. Это нужно, чтобы в будущем можно было менять провайдера хранения.

## 🧩 MediaPaths

`MediaPaths` — единая точка описания структуры Storage.

Текущие и будущие пути:

```text
user_avatars/{userId}/avatar.jpg
group_avatars/{chatId}/avatar.jpg
attachments/{chatId}/{messageId}/{fileName}
previews/{chatId}/{messageId}/{fileName}
```

Зачем это нужно:

- не размазывать строки путей по всему проекту;
- упростить миграцию Storage;
- упростить изменение структуры файлов;
- избежать разных вариантов одного и того же пути;
- централизованно контролировать архитектуру хранения.

## 🧠 Стратегия хранения медиа

На текущем этапе Firebase Storage используется рационально и ограниченно.

### Сейчас в Firebase Storage планируется хранить

- пользовательские аватары;
- групповые аватары;
- небольшие системные изображения;
- будущие превью.

### В будущем нельзя бесконтрольно хранить

- тяжёлые фото без сжатия;
- видео;
- голосовые сообщения без сжатия;
- крупные документы;
- временные файлы без очистки.

### Будущая гибридная архитектура

Возможные направления:

- Firebase Storage для системных медиа и превью;
- локальный кэш на устройстве;
- Google Drive / Яндекс Диск / S3 / собственный backend для тяжёлых вложений;
- Firestore только для метаданных и ссылок;
- автоматическая очистка кэша;
- ограничения размеров файлов;
- генерация thumbnails/previews.

## 🖼 Аватары и кэширование

Будущая система аватаров должна работать так:

```text
Пользователь выбирает изображение
    ↓
Кроп / сжатие до загрузки
    ↓
Загрузка в Storage
    ↓
Сохранение MediaAsset / avatar metadata в Firestore
    ↓
UI показывает аватар
    ↓
Клиент кэширует изображение локально
    ↓
Если avatarVersion не изменился — повторно не скачивает
```

Планируемые поля:

```text
avatar:
  provider
  path
  version
  updatedAt
  sizeBytes
  contentType
```

или совместимый вариант с существующими полями профиля.

Цель:

- не загружать оригиналы из галереи;
- уменьшать изображения до разумного размера;
- не скачивать один и тот же аватар повторно;
- снижать Storage download operations;
- снижать egress;
- контролировать расходы.

## 🛡 Firestore и Storage Rules

Проект использует Firebase Security Rules как обязательный уровень защиты.

Firestore Rules отвечают за:

- доступ к пользователям;
- доступ к чатам;
- доступ к сообщениям;
- проверку членства в чатах;
- ограничения ролей;
- права на отправку сообщений;
- защиту от удаления данных;
- защиту последнего администратора.

Storage Rules отвечают за:

- доступ к файлам;
- ограничения по владельцу;
- ограничения размера;
- MIME type;
- запрет неизвестных путей;
- будущие правила для аватаров, групповых аватаров и вложений.

Все правила должны храниться в корне проекта:

```text
firestore.rules
firestore.indexes.json
storage.rules
firebase.json
```

Инфраструктурные правила не должны лежать внутри `lib/`.

## 🧩 Spaces

Spaces — стратегическое направление развития Epistola.

Spaces не являются обычными группами или чатами. Это будущая платформа рабочих мини-приложений.

Планируемые модули:

- график смен;
- рабочие задачи;
- объявления;
- документы;
- журналы;
- заявки;
- рабочие панели;
- персональные и групповые дашборды.

Архитектурная идея Spaces:

- администраторы управляют доступными модулями пространства;
- пользователь сам включает или скрывает модули для себя;
- Space становится персонализированной рабочей панелью;
- модули могут развиваться независимо;
- в будущем Spaces могут стать основой корпоративной платформы, а не только частью мессенджера.

## 🎯 Принципы разработки

Проект развивается по следующим принципам:

- Архитектура важнее скорости разработки.
- Сначала проектирование — потом реализация.
- Каждый крупный модуль сначала проектируется как отдельная подсистема.
- UI не должен знать детали инфраструктуры.
- Firebase считается текущей инфраструктурой, но не вечной зависимостью.
- Domain-модели не должны зависеть от Firebase, Flutter или UI.
- Firestore хранит данные и метаданные, но не сами файлы.
- Медиафайлы должны проходить сжатие и ограничения размеров.
- Все пути Storage должны быть централизованы.
- После каждого крупного этапа обновляется документация.
- После каждой стабильной версии создаётся Git Tag.
- Код должен проходить `flutter analyze` без ошибок.
- Для Firebase CLI на Windows используются команды через `firebase.cmd`.

## 🧭 Стратегия контроля расходов

Epistola проектируется с учётом ограниченного бюджета инфраструктуры.

Особое внимание:

- Firestore reads/writes;
- Storage download operations;
- Storage egress;
- размер аватаров;
- повторная загрузка одних и тех же изображений;
- локальный кэш;
- сжатие изображений;
- ограничения размеров файлов;
- мониторинг реального потребления.

На этапе пилота важно не оптимизировать вслепую, а собрать реальные метрики использования. По итогам анализа можно будет принять решение о регионе Storage, backend-хранилище, кэшировании и дальнейшей архитектуре медиа.

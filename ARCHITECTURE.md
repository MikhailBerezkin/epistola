## Назначение

Epistola — современный корпоративный мессенджер, создаваемый на Flutter с использованием Firebase.

Первоначально проект ориентирован на использование внутри предприятия, однако архитектура изначально проектируется как универсальная платформа, пригодная для дальнейшего масштабирования и использования в организациях любого размера.

Основной принцип разработки — создание системы, которая не потребует полного переписывания при росте функциональности.

---

## Долгосрочная цель

Главная цель проекта — создать единую рабочую платформу предприятия.

В долгосрочной перспективе Epistola должна объединить:

- корпоративный мессенджер;
- рабочие пространства (Spaces);
- мини-приложения;
- документы;
- рабочие задачи;
- сменные графики;
- внутренние сервисы компании;
- инструменты взаимодействия подразделений.

Таким образом Epistola должна стать не только средством обмена сообщениями, но и основной цифровой рабочей средой организации.

## Media Storage Strategy

Firebase Storage подключён, но должен использоваться экономно.

На первом этапе Firebase Storage используется для лёгких и системных медиа:

- аватары пользователей;
- аватары групп;
- изображения, связанные с профилями и карточками;
- небольшие превью.

Для тяжёлых медиа требуется отдельное архитектурное решение:

- фотографии;
- видео;
- голосовые сообщения;
- документы;
- крупные вложения.

Перед загрузкой медиа в облако необходимо предусмотреть:

- сжатие изображений;
- сжатие видео;
- сжатие голосовых сообщений;
- генерацию превью;
- ограничения по размеру файлов;
- локальное кеширование;
- очистку старых локальных файлов.

В будущем нужно рассмотреть гибридную модель хранения:

- Firebase Storage — для системных и небольших файлов;
- устройство пользователя — для локального хранения и кеша;
- внешние облака, например Google Drive или Яндекс Диск — для тяжёлых файлов и документов.

Firestore при этом должен хранить не сами файлы, а только метаданные:

- тип файла;
- размер;
- имя;
- владелец;
- chatId;
- downloadUrl или externalUrl;
- createdAt;
- storageProvider.

# Философия проекта

## Основные принципы

Epistola разрабатывается не как учебный проект и не как экспериментальный мессенджер.

Архитектура изначально проектируется таким образом, чтобы приложение могло непрерывно развиваться в течение многих лет без необходимости полного переписывания.

При принятии любых технических решений используются следующие принципы.

---

## Архитектура важнее скорости разработки

Самый главный принцип проекта.

Перед реализацией нового функционала сначала проектируется архитектура, после чего начинается написание кода.

Временные решения допускаются только тогда, когда они не ухудшают дальнейшее развитие проекта.

Если существует выбор между быстрым решением и правильной архитектурой, предпочтение всегда отдаётся архитектуре.

---

## Масштабируемость

Каждая новая возможность должна разрабатываться с расчётом на дальнейшее расширение.

Новая функциональность не должна ограничивать развитие проекта через несколько месяцев.

При проектировании всегда учитываются будущие версии Epistola.

---

## Разделение ответственности

Каждый компонент проекта отвечает только за собственную область.

Например:

- экран отвечает только за отображение;
- сервис отвечает только за бизнес-логику;
- Firebase отвечает только за хранение данных;
- модель описывает только структуру данных.

Пересечение ответственности считается архитектурной ошибкой.

---

## Минимизация дублирования

Любой повторяющийся код постепенно выносится:

- в отдельный сервис;
- в отдельный виджет;
- в helper;
- в модель данных.

Копирование логики допускается только как временная мера во время рефакторинга.

---

## Простота решений

Если задачу можно решить двумя способами, предпочтение отдаётся наиболее простому и понятному.

Архитектура не должна усложняться без объективной причины.

Любое усложнение должно приносить реальную пользу проекту.

---

## Последовательное развитие

Epistola развивается небольшими законченными этапами.

Каждый этап должен:

- иметь понятную цель;
- завершаться полностью;
- проходить тестирование;
- сопровождаться обновлением документации;
- завершаться Git-тегом (для стабильных релизов).

Это позволяет всегда иметь рабочую стабильную версию проекта.

# Архитектурные решения

Данный раздел фиксирует ключевые решения, которые считаются фундаментальными для проекта.

Эти решения не должны изменяться без серьёзной необходимости.

---

## Firebase Authentication используется только для авторизации

Firebase Authentication отвечает исключительно за вход пользователя в систему.

Он не используется как источник данных профиля.

В профиле пользователя не должны храниться данные, напрямую зависящие от Firebase Authentication.

Это позволяет в будущем безболезненно добавить:

- вход по телефону;
- Google Sign-In;
- Microsoft;
- Apple;
- другие способы авторизации.

---

## Профиль пользователя хранится отдельно

Информация профиля располагается в коллекции `users`.

Публичный контактный E-mail не зависит от E-mail авторизации.

Телефон также является обычным полем профиля.

Такое разделение делает профиль независимым от механизма входа в систему.

---

## UID является главным идентификатором

Во всём проекте основным идентификатором пользователя считается только Firebase UID.

E-mail используется исключительно как дополнительная информация.

Никакая критическая логика не должна строиться только на адресах электронной почты.

---

## Firestore — источник истины

Главное состояние приложения всегда хранится в Firestore.

Интерфейс отображает данные Firestore, но не является источником истины.

Это обеспечивает:

- синхронизацию нескольких устройств;
- корректную работу нескольких пользователей;
- возможность дальнейшего масштабирования.

---

## Firestore Rules являются частью архитектуры

Правила безопасности Firestore рассматриваются как полноценная часть серверной логики.

Проверки, влияющие на безопасность приложения, не должны выполняться только пользовательским интерфейсом.

Даже модифицированный клиент не должен иметь возможности обойти ограничения ролей или модерации.

---

## ChatService является фасадом

Пользовательский интерфейс взаимодействует только с `ChatService`.

Внутреннее устройство сервисов может изменяться без изменения экранов приложения.

Такой подход обеспечивает стабильный публичный API и упрощает дальнейший рефакторинг.

---

## Большие сервисы разделяются

При увеличении размера сервисов они разделяются на специализированные компоненты.

На текущем этапе реализованы:

- ChatMessagesService
- ChatGroupsService
- ChatMembersService
- ChatPermissionsService
- ChatPrivateService
- ChatSearchService

В дальнейшем аналогичный подход будет использоваться и для других крупных подсистем.

---

## Один файл — одна ответственность

Каждый файл проекта должен иметь понятную область ответственности.

Если файл начинает содержать несколько независимых направлений логики, он должен быть разделён.

Монолитные файлы допускаются только на ранних этапах разработки.

---

## UI не содержит бизнес-логики

Экран отвечает только за:

- отображение данных;
- обработку пользовательских действий;
- вызов соответствующего сервиса.

Работа с Firebase, ролями, модерацией, поиском и сообщениями выполняется исключительно сервисами.

---

## Новые функции должны расширять архитектуру

Каждый новый этап разработки должен усиливать существующую архитектуру, а не нарушать её.

Приоритет всегда отдаётся расширению уже существующих механизмов, а не созданию альтернативных решений.

---

# Архитектура приложения

Epistola построена по принципу многоуровневой архитектуры.

Каждый уровень отвечает исключительно за собственную область ответственности.

На текущем этапе используется следующая схема.

```text
                           Пользователь

                                │

                                ▼

                        Flutter UI Layer

                Screens / Widgets / Dialogs

                                │

                                ▼

                     Application Services

                          ChatService

                                │

     ┌───────────────┬───────────────┬───────────────┐

     ▼               ▼               ▼

 Messages       Groups & Roles     Search

     │               │               │

     └───────────────┴───────────────┐

                                     ▼

                            Firebase Layer

                   Authentication / Firestore

                                     │

                                     ▼

                         Firebase Storage

                          (будущий этап)
```

Архитектура специально проектируется таким образом, чтобы пользовательский интерфейс не зависел напрямую от структуры Firebase.

Это позволит без серьёзных изменений масштабировать проект в будущем, а также добавлять новые сервисы без изменения существующих экранов.

# Структура проекта

## Общая структура

Исходный код приложения располагается в каталоге `lib`.

Проект развивается по принципу разделения ответственности. Каждый каталог отвечает за собственную область и не должен дублировать функциональность других частей системы.

Текущая структура проекта:

```text
lib/

├── helpers/
├── models/
├── screens/
├── services/
│
│   └── chat/
│
├── theme/
├── widgets/
│
├── firebase_options.dart
└── main.dart
```

---

## helpers/

Каталог содержит небольшие вспомогательные классы и функции.

Helpers используются тогда, когда логика:

- не относится к бизнес-процессам;
- не требует собственного сервиса;
- используется сразу в нескольких местах проекта.

Типичные примеры:

- преобразование ролей;
- форматирование времени;
- отображение статусов;
- небольшие утилиты.

Если helper начинает содержать сложную бизнес-логику, он должен быть преобразован в полноценный сервис.

---

## models/

Каталог моделей данных.

Каждая модель должна описывать только структуру объекта.

Модель не должна содержать Firebase-запросы, навигацию или пользовательский интерфейс.

На текущем этапе используются модели пользователей и сообщений.

В дальнейшем здесь будут располагаться:

- AppUser
- Chat
- Message
- MessageAttachment
- Space
- SpaceModule
- Shift
- Task
- Announcement

Все модели должны поддерживать сериализацию в Firestore и обратно.

---

## screens/

Полноценные страницы приложения.

Каждый экран отвечает только за отображение информации и взаимодействие с пользователем.

Бизнес-логика должна находиться в сервисах.

Типичные экраны:

- WelcomeScreen
- LoginScreen
- RegisterScreen
- HomeScreen
- ChatsPage
- SpacesPage
- ProfilePage
- ChatScreen
- ContactsScreen
- UserSearchScreen

- EditProfileScreen

При появлении новых функций рекомендуется создавать отдельные экраны вместо перегрузки существующих.

---

## widgets/

Каталог переиспользуемых компонентов интерфейса.

Главная задача виджетов — уменьшить размер экранов и исключить копирование кода.

Примеры:

- ChatTile
- MessageBubble
- MessageInput
- MessagesList
- GroupHeader
- ProfileHeader
- FoundUserCard
- ContactTile

Каждый виджет должен решать одну конкретную задачу.

---

## theme/

Содержит оформление приложения.

Здесь располагаются:

- светлая тема;
- тёмная тема;
- общие цвета;
- стили текста;
- настройки Material 3.

В дальнейшем здесь будут находиться пользовательские темы оформления.

# Сервисная архитектура

## Общая концепция

Практически вся бизнес-логика Epistola находится в сервисах.

Экран не должен напрямую обращаться к Firebase, если соответствующая логика уже существует в сервисном слое.

Это позволяет:

- уменьшить связанность проекта;
- облегчить тестирование;
- безопасно выполнять рефакторинг;
- использовать одну и ту же бизнес-логику несколькими экранами.

---

## ChatService

ChatService является публичным фасадом системы сообщений.

Все экраны работают именно с ним.

Внутри ChatService вызовы распределяются между специализированными сервисами.

Такой подход позволяет менять внутреннюю архитектуру без изменения пользовательского интерфейса.

---

## ChatBaseService

Базовый сервис.

Предоставляет наследникам:

- FirebaseAuth;
- FirebaseFirestore;
- общие методы;
- общие проверки.

Все специализированные сервисы наследуются именно от него.

---

## ChatMessagesService

Отвечает исключительно за сообщения.

Основные обязанности:

- получение списка сообщений;
- отправка сообщений;
- отметка сообщений как прочитанных;
- подсчёт непрочитанных сообщений;
- обновление последнего сообщения чата;
- автоматическое снятие временных ограничений после окончания срока действия.

В будущем сюда будет перенесена работа со всеми типами вложений.

---

## ChatGroupsService

Отвечает за жизненный цикл групп.

Основные функции:

- создание группы;
- изменение названия;
- выход участников;
- безопасный выход администратора;
- передача прав;
- роспуск группы;
- управление составом группы.

---

## ChatMembersService

Работает исключительно с участниками.

Основные функции:

- получение участников;
- поиск пользователей;
- добавление участников;
- изменение ролей;
- получение информации о пользователе.

---

## ChatPermissionsService

Отвечает за права доступа.

Содержит всю бизнес-логику:

- ролей;
- mute;
- ban;
- guest;
- разрешений на отправку сообщений;
- ограничений группы.

Все изменения прав должны проходить через этот сервис.

---

## ChatPrivateService

Полностью отвечает за личные чаты.

Основные задачи:

- поиск существующего личного диалога;
- создание нового;
- восстановление старых структур данных при необходимости;
- поддержание единого алгоритма формирования private chat.

В проекте используется принцип:

> между двумя пользователями существует только один личный чат.

---

## ChatSearchService

Отвечает за поиск.

Поддерживает:

- поиск пользователей;
- поиск по имени;
- поиск по телефону;
- поиск по публичному email;
- получение списка пользователей;
- получение списка чатов.

Поиск должен быть максимально независим от экранов.

---

# Правила развития сервисов

При появлении нового крупного функционала рекомендуется создавать отдельный специализированный сервис.

Например:

```text
AttachmentService

StorageService

NotificationService

SpaceService

ShiftService

TaskService
```

Это предпочтительнее, чем бесконечно увеличивать существующие классы.

---

# Правила именования

Для всех новых сервисов используются единые соглашения.

Базовый сервис:

```text
SomethingBaseService
```

Основной сервис:

```text
SomethingService
```

Специализированные сервисы:

```text
SomethingMessagesService

SomethingStorageService

SomethingPermissionsService
```

Названия должны максимально точно отражать область ответственности.

---

# Правила экранов

Каждый экран обязан соблюдать следующие требования.

Экран:

- отображает информацию;
- вызывает сервисы;
- обрабатывает пользовательские действия.

Экран не должен:

- выполнять сложные вычисления;
- содержать Firebase-запросы;
- хранить бизнес-логику;
- дублировать код других экранов.

Все подобные задачи должны переноситься в сервисный слой.

---
# Cloud Firestore

## Общая концепция

Cloud Firestore является основным источником данных Epistola.

Практически всё состояние приложения хранится именно в Firestore.

Пользовательский интерфейс не считается источником истины и всегда отображает актуальное состояние базы данных.

Такой подход обеспечивает:

- синхронизацию нескольких устройств;
- работу нескольких пользователей одновременно;
- поддержку потоковых обновлений;
- возможность дальнейшего масштабирования.

Каждый документ соответствует одному пользователю Firebase Authentication.

Основные поля:


В дальнейшем будут добавлены:

```text

department

language

notificationSettings
```

## Коллекция chats

Каждый документ представляет отдельный чат.

Поддерживаются два типа:

```text

```

Основные поля:

```text
type

name

memberIds

memberEmails

memberRoles

memberStatus

groupSettings

lastMessage

lastMessageAt

lastRead

```

Все изменения состава участников должны выполняться исключительно сервисным слоем.

---

## Подколлекция messages

Каждый чат содержит собственную коллекцию сообщений.

Структура:

```text

```

# Firestore Rules

Безопасность Epistola строится не только на пользовательском интерфейсе.

Основной уровень защиты реализован именно через Firestore Rules.

Даже модифицированное клиентское приложение не должно иметь возможности:

- изменить собственную роль;
- снять собственный бан;
- писать сообщения при отсутствии прав;
- читать чужие чаты;
- получить доступ к закрытым данным.

Все критические проверки должны существовать одновременно:

- в пользовательском интерфейсе;
- в сервисном слое;
- в Firestore Rules.

---

## Роли

Поддерживаются роли:

```text
owner

admin

moderator

member

guest
```

Каждая роль определяет доступные действия пользователя внутри конкретной группы.

---

## Статусы

Поддерживаются статусы:

```text
normal

muted

banned
```

Статус является отдельной сущностью и не зависит от роли пользователя.

Например:

Администратор может временно получить статус `muted`.

---

## Message Permission

Каждая группа хранит собственные настройки отправки сообщений.

Поддерживаются режимы:

```text
all

moderators

admins
```

Настройки проверяются одновременно:

- пользовательским интерфейсом;
- сервисами;
- Firestore Rules.

---

# Media Foundation

## Назначение

Media Foundation является следующим крупным этапом развития Epistola.

Главная цель — создать единую архитектуру хранения любых вложений сообщений.

После завершения данного этапа дальнейшее добавление новых типов файлов не должно требовать изменения общей архитектуры.

---

## Поддерживаемые типы

Планируется поддержка:

- изображения;
- фотографии;
- документы;
- видео;
- голосовые сообщения;
- архивы;
- произвольные файлы.

---

## Архитектура

Сообщение перестаёт быть исключительно текстовым.

Каждое сообщение сможет содержать:

```text
Text

+

Attachments
```

Количество вложений не должно ограничиваться архитектурой модели.

---

## MessageAttachment

Каждое вложение должно описываться отдельной моделью.

Планируемые поля:

```text
id

type

name

mimeType

storagePath

downloadUrl

thumbnailUrl

size

duration

width

height

createdAt
```

Такая модель позволит работать со всеми типами файлов одинаковым способом.

---

## Firebase Storage

Все файлы должны храниться исключительно в Firebase Storage.

Cloud Firestore хранит только:

- описание;
- ссылку;
- метаданные.

Это значительно уменьшает размер документов Firestore.

---

## Attachment Menu

Возле поля ввода сообщений используется кнопка "+"

Она открывает нижнюю панель действий.

Планируемые действия:

- Фото
- Камера
- Видео
- Документ
- Файл
- Контакт
- Геопозиция

Архитектура меню должна позволять без изменений интерфейса добавлять новые действия.

---
# Spaces

## Общая концепция

Spaces являются стратегическим направлением развития проекта Epistola.

Несмотря на внешнее сходство с группами, **Spaces не являются чатами**.

Это отдельная архитектурная подсистема, предназначенная для организации рабочих процессов предприятия.

Главная задача Spaces — объединить общение сотрудников и рабочие инструменты внутри единой среды.

Чат является лишь одним из возможных модулей Space.

---

## Основная идея

Space представляет собой рабочее пространство.

Рабочее пространство может соответствовать:

- подразделению;
- отделу;
- смене;
- бригаде;
- проекту;
- объекту;
- филиалу;
- любой другой организационной единице.

Каждый Space обладает собственным набором доступных модулей.

---

## Spaces не являются группами

Это одно из фундаментальных архитектурных решений проекта.

Группа предназначена исключительно для общения пользователей.

Space является рабочей платформой.

Следовательно:

- группа может существовать без Space;
- Space может содержать несколько чатов;
- чат может быть только одним из модулей Space.

Данные сущности не должны объединяться.

---

## Модульная архитектура

Каждый Space состоит из независимых модулей.

Примеры модулей:

- Chat
- Shift Schedule
- Tasks
- Announcements
- Documents
- Contacts
- Calendar
- Knowledge Base
- Reports
- Forms

Архитектура должна позволять подключать новые модули без изменения существующих.

---

## Доступные модули

Каждый Space хранит перечень модулей, разрешённых администраторами.

Например:

```text
Available Modules

✓ Chat

✓ Schedule

✓ Tasks

✓ Documents

✓ Announcements

✗ Reports
```

Пользователь не может самостоятельно использовать модуль, который отсутствует в списке доступных.

---

## Персональные модули

Каждый пользователь хранит собственный список включённых модулей.

Например:

```text
Enabled Modules

✓ Chat

✓ Schedule

✓ Tasks

✗ Documents

✗ Announcements
```

Таким образом пользователь самостоятельно настраивает собственное рабочее пространство.

Отключение модуля скрывает его только для данного пользователя.

Остальные участники продолжают работать без изменений.

---

## Dashboard

В дальнейшем Space должен открываться не списком модулей, а персональной рабочей панелью.

Dashboard может содержать:

- сегодняшнюю смену;
- ближайшие задачи;
- новые документы;
- последние объявления;
- непрочитанные сообщения;
- быстрые действия.

Каждый пользователь сможет настраивать Dashboard под собственные задачи.

---

## Архитектурные преимущества

Такой подход обеспечивает:

- масштабируемость;
- персонализацию интерфейса;
- независимость модулей;
- возможность разработки новых функций без изменения существующих.

# Release Process

Каждый стабильный релиз проходит одинаковый цикл.

1. Завершение функциональности.
2. Тестирование.
3. Проверка `flutter analyze`.
4. Сборка Release APK.
5. Проверка на устройстве.
6. Обновление README.md.
7. Обновление ARCHITECTURE.md.
8. Commit.
9. Push.
10. Git Tag.
11. Проверка GitHub.

Нарушение данной последовательности не рекомендуется.

# Правила разработки

При дальнейшем развитии проекта должны соблюдаться следующие правила.

- Не ломать существующую архитектуру без серьёзной причины.
- Новая функциональность должна расширять существующую систему.
- Один файл — одна ответственность.
- UI не содержит бизнес-логики.
- Firestore Rules обязательны для всех критических операций.
- Firebase вызывается преимущественно через сервисы.
- Документация обновляется одновременно с кодом.
- После стабильной версии создаётся Git Tag.
- Для Firebase CLI на Windows используются команды через `firebase.cmd`.

---

# Неприкосновенные архитектурные решения

Следующие решения считаются фундаментальными.

Их изменение требует серьёзного архитектурного обоснования.

- Firebase Authentication используется только для авторизации.
- Профиль пользователя хранится отдельно в Firestore.
- UID является единственным основным идентификатором пользователя.
- Firestore является источником истины.
- Firestore Rules являются обязательной частью безопасности.
- ChatService остаётся фасадом сервисной архитектуры.
- Большие сервисы разделяются по зонам ответственности.
- UI не содержит бизнес-логики.
- Spaces не являются группами.
- Чат является лишь одним из модулей Space.
- Пользователь самостоятельно управляет отображением доступных модулей своего Space.
- Файлы хранятся в Firebase Storage, а не в Firestore.

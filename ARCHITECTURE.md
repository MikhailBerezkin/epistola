# 1. О проекте

## 1.1 Назначение

**Epistola** — современный корпоративный мессенджер, разрабатываемый на Flutter с использованием Firebase.

Первоначально проект создаётся для внутреннего использования в компании, однако архитектура изначально проектируется с расчётом на дальнейшее развитие в полноценную коммуникационную платформу.

---

## 1.2 Основные цели

Основные цели проекта:

- обеспечить удобное общение сотрудников;
- заменить разрозненные корпоративные чаты;
- предоставить удобную систему управления рабочими группами;
- создать безопасную систему ролей и модерации;
- обеспечить дальнейшее расширение без переписывания архитектуры.

---

## 1.3 Долгосрочное видение

Epistola не ограничивается ролью обычного мессенджера.

Долгосрочная цель проекта — создание единой рабочей среды, объединяющей коммуникацию сотрудников и рабочие инструменты внутри одной платформы.

В будущем ядром системы станут **Пространства (Spaces)** — настраиваемые рабочие области с подключаемыми мини-приложениями.

Планируемые направления развития:

- корпоративные чаты;
- рабочие пространства (Spaces);
- модульное расширение функциональности;
- совместная работа сотрудников;
- уведомления;
- хранение файлов;
- инструменты управления подразделениями;
- мобильная и веб-версии.

# 2. Философия проекта

## 2.1 Основные принципы

При разработке Epistola используются следующие принципы.

### Архитектура важнее скорости

Перед реализацией нового функционала сначала продумывается архитектура, затем пишется код.

Временные решения допускаются только в исключительных случаях.

---

### Простота важнее сложности

Предпочтение отдаётся простым и понятным решениям.

Если задача может быть реализована без усложнения архитектуры, выбирается именно этот путь.

---

### Масштабируемость

Каждый новый компонент должен легко расширяться в будущем без полного переписывания проекта.

---

### Минимизация дублирования

Повторяющийся код постепенно выносится в отдельные сервисы, виджеты и вспомогательные классы.

---

### Единый стиль проекта

Все новые экраны, сервисы и компоненты должны соответствовать единому стилю Epistola.

Изменения не должны создавать ощущение, что различные части приложения написаны разными разработчиками.

---

## 2.2 UX-философия

При проектировании интерфейса используются следующие принципы:

- минимализм;
- Material 3;

# 3. Архитектура приложения

## 3.1 Общая архитектура

Epistola построена по принципу разделения ответственности (Separation of Responsibilities).

Каждый уровень приложения отвечает только за собственную область логики.

На текущем этапе архитектура выглядит следующим образом:

```text
                    Пользователь

                          │
                          ▼

                    Flutter UI
        (Screens / Widgets / Dialogs)

                          │
                          ▼

                    ChatService
              (единая публичная точка)

                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ▼                 ▼                 ▼

 ChatMessages      ChatGroups       ChatMembers

        │                 │                 │
        └──────────────┬──┴──────────────┬──┘
                       │                 │
                       ▼                 ▼

            ChatPermissions      ChatPrivate

                       │
                       ▼

                Firebase Firestore

                       │
                       ▼

                  Firebase Storage
                     (Roadmap)
```

---

## 3.2 Основные уровни

### UI

Экранный слой приложения.

Содержит:

- Screens;
- Widgets;
- BottomSheet;
- Dialog;
- Navigation.

UI не содержит бизнес-логики и по возможности не взаимодействует напрямую с Firebase.

---

### Services

Сервисный слой содержит всю бизнес-логику приложения.

Именно здесь выполняются:

- работа с чатами;
- работа с участниками;
- работа с ролями;
- работа с сообщениями;
- работа с поиском;
- взаимодействие с Firebase.

Большинство экранов взаимодействует только с сервисами.

---

### Firebase

Firebase является слоем хранения данных.

Используются:

- Firebase Authentication;
- Cloud Firestore.

В следующих версиях будут добавлены:

- Firebase Storage;
- Firebase Cloud Messaging (FCM).

---

## 3.3 ChatService

После рефакторинга версии **v0.5.10** ChatService выполняет роль фасада (Facade).

Он предоставляет единый публичный API для всего приложения.

Внутри ChatService логика распределяется между специализированными сервисами.

Это позволяет:

- уменьшить размер файлов;
- изолировать ответственность;
- упростить сопровождение;
- безопасно расширять функциональность.

---

## 3.4 Специализированные сервисы

В настоящее время архитектура разделена следующим образом.

| Сервис | Назначение |
|---------|------------|
| ChatMessagesService | Работа с сообщениями |
| ChatGroupsService | Управление группами |
| ChatMembersService | Работа с участниками |
| ChatPermissionsService | Роли, мьюты, баны, права |
| ChatPrivateService | Личные диалоги |
| ChatSearchService | Поиск пользователей и чатов |

Все сервисы наследуются от `ChatBaseService`, который предоставляет общий доступ к Firebase Authentication и Cloud Firestore.

---

## 3.5 Принципы развития архитектуры

При добавлении нового функционала предпочтение отдаётся созданию нового специализированного сервиса вместо чрезмерного увеличения существующих файлов.

При этом публичный API `ChatService` должен оставаться максимально стабильным, чтобы изменения внутренней архитектуры не требовали переработки пользовательского интерфейса.

# 4. Структура проекта

## 4.1 Общая структура

Исходный код приложения расположен в каталоге `lib`.

Каждый каталог отвечает за собственную область ответственности.

Текущая структура проекта выглядит следующим образом.

```text
lib/

├── helpers/
├── models/
├── screens/
├── services/
│
│   └── chat/
│
├── theme/
├── widgets/
│
├── firebase_options.dart
└── main.dart
```

---

## 4.2 Назначение каталогов

### helpers/

Вспомогательные классы и функции.

Используются для небольшой переиспользуемой логики, которая не относится к бизнес-сервисам.

Примеры:

- RoleHelper
- StatusHelper

---

### models/

Модели данных приложения.

Каждая модель описывает структуру объектов, используемых в приложении.

Примеры:

- AppUser

В дальнейшем здесь также будут располагаться модели:

- Chat
- Message
- Space
- Attachment

---

### screens/

Полноценные страницы приложения.

Каждый экран представляет отдельную страницу навигации Flutter.

Например:

- LoginScreen
- HomeScreen
- ChatScreen
- GroupInfoScreen
- SettingsScreen

Экраны отвечают исключительно за отображение интерфейса и взаимодействие с пользователем.

Основная бизнес-логика должна находиться в сервисах.

---

### services/

Сервисный слой приложения.

Именно здесь располагается практически вся бизнес-логика Epistola.

Сервисы выполняют:

- работу с Firebase;
- создание чатов;
- поиск пользователей;
- отправку сообщений;
- управление участниками;
- модерацию;
- права доступа.

После рефакторинга версии **v0.5.10** сервисы разделены по областям ответственности.

---

### services/chat/

Специализированные сервисы работы с чатами.

Структура каталога:

```text
chat/

├── chat_base_service.dart
├── chat_messages_service.dart
├── chat_groups_service.dart
├── chat_members_service.dart
├── chat_permissions_service.dart
├── chat_private_service.dart
└── chat_search_service.dart
```

---

### ChatBaseService

Базовый сервис.

Предоставляет наследникам общий доступ к:

- FirebaseAuth;
- Cloud Firestore.

Это исключает дублирование одинакового кода во всех сервисах.

---

### ChatMessagesService

Отвечает за сообщения.

Основные обязанности:

- отправка сообщений;
- получение сообщений;
- отметка прочитанного;
- подсчёт непрочитанных сообщений;
- автоматическое снятие временных ограничений после окончания срока действия.

---

### ChatGroupsService

Отвечает за управление группами.

Основные обязанности:

- создание группы;
- выход из группы;
- безопасный выход последнего администратора;
- передача прав;
- роспуск группы;
- подсчёт администраторов.

---

### ChatMembersService

Отвечает за участников.

Основные обязанности:

- получение участников;
- получение пользователей по UID;
- поиск пользователей вне группы;
- добавление участников;
- изменение ролей.

---

### ChatPermissionsService

Отвечает за систему прав.

Основные обязанности:

- изменение разрешений группы;
- выдача mute;
- снятие mute;
- выдача ban;
- снятие ban;
- автоматическое снятие временных ограничений.

---

### ChatPrivateService

Отвечает за личные диалоги.

Основные обязанности:

- поиск существующего личного чата;
- создание личного чата;
- восстановление структуры старых диалогов при необходимости.

---

### ChatSearchService

Отвечает за поиск.

Основные обязанности:

- поиск пользователей;
- поиск по телефону;
- поиск по электронной почте;
- получение списка чатов пользователя;
- получение списка пользователей.

---

### theme/

Оформление приложения.

Здесь находятся:

- светлая тема;
- тёмная тема;
- настройки оформления.

---

### widgets/

Переиспользуемые элементы интерфейса.

В отличие от экранов, виджеты используются сразу в нескольких местах приложения.

После рефакторинга большая часть интерфейса состоит именно из небольших независимых компонентов.

Примеры:

- MessageBubble
- ChatTile
- MessageInput
- GroupHeader
- ProfileHeader
- FoundUserCard

---

### firebase_options.dart

Автоматически сгенерированная конфигурация Firebase.

Редактируется только средствами FlutterFire CLI.

---

### main.dart

Точка входа приложения.

Именно отсюда начинается запуск Epistola.

Основные задачи:

- инициализация Flutter;
- инициализация Firebase;
- запуск приложения;
- подключение темы оформления.

---

## 4.3 Принципы организации проекта

# 5. Firebase

## 5.1 Общая концепция

В качестве серверной платформы Epistola использует **Firebase**.

На текущем этапе Firebase обеспечивает:

- авторизацию пользователей;
- хранение данных;
- применение правил безопасности.

Архитектура проекта изначально строится таким образом, чтобы максимально использовать возможности Firebase без создания собственного серверного API.

---

## 5.2 Используемые сервисы Firebase

На текущий момент используются следующие сервисы.

| Сервис | Назначение |
|---------|------------|
| Firebase Authentication | Авторизация пользователей |
| Cloud Firestore | Основное хранилище данных |

В следующих версиях будут подключены:

| Сервис | Статус |
|---------|---------|
| Firebase Storage | Планируется (v0.6.x) |
| Firebase Cloud Messaging (FCM) | Планируется |
| Firebase Analytics | Возможно позднее |
| Crashlytics | Возможно позднее |

---

## 5.3 Firebase Authentication

Firebase Authentication используется исключительно для идентификации пользователя.

После входа приложение получает:

- UID пользователя;
- адрес электронной почты;
- информацию о текущей сессии.

UID является главным идентификатором пользователя во всей системе.

Электронная почта используется только для авторизации и не считается публичной контактной информацией.

---

## 5.4 Пользовательский профиль

Профиль пользователя хранится отдельно от Firebase Authentication.

Это сделано намеренно.

Причины такого решения:

- возможность изменить публичный контактный E-mail без изменения входа в систему;
- независимость профиля от механизма авторизации;
- возможность использовать различные способы входа в будущем (телефон, Google, Microsoft и др.);
- разделение служебных и пользовательских данных.

В результате:

Firebase Authentication отвечает только за вход в систему.

Cloud Firestore хранит всю информацию профиля пользователя.

---

## 5.5 Cloud Firestore

Cloud Firestore является основным хранилищем данных проекта.

В Firestore располагаются:

- пользователи;
- чаты;
- сообщения;
- роли;
- статусы участников;
- настройки групп;
- информация о прочтении сообщений.

Большинство данных приложения хранится именно здесь.

---

## 5.6 Firestore как источник истины

Epistola использует модель, при которой именно Firestore является главным источником актуального состояния системы.

Пользовательский интерфейс отображает данные, полученные из Firestore, и не считается источником истины.

Это обеспечивает:

- корректную синхронизацию между устройствами;
- единое состояние данных;
- возможность масштабирования;
- поддержку работы нескольких пользователей одновременно.

---

## 5.7 Firestore Rules

Безопасность проекта строится не только на ограничениях пользовательского интерфейса.

Все критически важные проверки должны подтверждаться правилами Firestore.

Даже если пользователь изменит клиентское приложение, Firestore Rules не позволят нарушить ограничения безопасности.

Именно поэтому роли, права, модерация и доступ к сообщениям проверяются как на стороне клиента, так и серверными правилами Firestore.

---

## 5.8 Firebase Storage (Roadmap)

Начиная с версии **v0.6.x** планируется подключение Firebase Storage.

Storage станет единым хранилищем медиафайлов приложения.

Планируется поддержка:

- изображений;
- документов;
- файлов;
- голосовых сообщений;
- пользовательских аватаров;
- групповых аватаров.

Каждый файл будет храниться отдельно от Firestore, а в базе данных будут сохраняться только его метаданные и ссылка.

Такой подход позволит значительно уменьшить объём документов Firestore и повысить производительность приложения.

---

## 5.9 Firebase Cloud Messaging (Roadmap)

После завершения Media Foundation планируется интеграция Firebase Cloud Messaging (FCM).

FCM будет использоваться для:

- push-уведомлений;
- уведомлений о новых сообщениях;
- уведомлений о приглашениях в группы;
- системных уведомлений;
- будущих уведомлений рабочих пространств (Spaces).

Архитектура приложения уже проектируется с учётом будущего подключения FCM.

## 6.1 Общая концепция

Cloud Firestore является основным хранилищем данных Epistola.

Практически всё состояние приложения хранится именно в Firestore.

Это позволяет:

- синхронизировать данные между устройствами;
- получать обновления в реальном времени;
- поддерживать офлайн-кэширование;
- масштабировать приложение без собственного серверного API.

---

## 6.2 Основные коллекции

На текущем этапе используются три основные коллекции.

```text
Firestore

users/

chats/

messages/
```

В действительности сообщения располагаются внутри документов чатов.

Полная структура выглядит следующим образом.

```text
users/

chats/
    chatId/
        messages/
```

---

## 6.3 Коллекция users

Содержит информацию обо всех зарегистрированных пользователях.

Каждый документ имеет идентификатор, совпадающий с UID пользователя Firebase Authentication.

```text
users/

uid
```

Типичные поля:

| Поле | Назначение |
|------|------------|
| uid | UID пользователя |
| name | Имя пользователя |
| email | Служебный E-mail (авторизация) |
| contactEmail | Публичный E-mail |
| phone | Телефон |
| about | Описание пользователя |
| avatarUrl | Ссылка на аватар |
| createdAt | Дата регистрации |

---

## 6.4 Коллекция chats

Каждый документ представляет отдельный чат.

Поддерживаются два типа:

- private
- group

Типичные поля:

| Поле | Назначение |
|------|------------|
| type | Тип чата |
| name | Название |
| memberIds | UID участников |
| memberEmails | E-mail участников |
| memberRoles | Роли |
| memberStatus | Статусы |
| groupSettings | Настройки группы |
| lastMessage | Последнее сообщение |
| lastMessageAt | Время последнего сообщения |
| lastRead | Последнее прочтение |
| isDissolved | Флаг роспуска группы |

---

## 6.5 Подколлекция messages

Каждый чат содержит собственную коллекцию сообщений.

```text
chats/

chatId/

messages/

messageId
```

Типичные поля:

| Поле | Назначение |
|------|------------|
| senderId | UID автора |
| senderEmail | E-mail автора |
| senderName | Имя автора |
| text | Текст сообщения |
| createdAt | Время отправки |

В следующих версиях структура сообщений будет расширена поддержкой медиафайлов.

---

## 6.6 Будущее развитие структуры

После реализации Media Foundation документы сообщений будут содержать дополнительные поля.

Например:

- attachments;
- imageUrl;
- thumbnailUrl;
- storagePath;
- mimeType;
- fileSize;
- duration (для голосовых сообщений).

Архитектура Firestore уже проектируется с учётом этого расширения.

---

## 6.7 Индексы

Для ускорения работы Firestore используются составные индексы.

На текущем этапе применяется индекс для получения списка пользовательских чатов.

Типичный запрос:

```dart
memberIds arrayContains currentUser.uid
orderBy(lastMessageAt, descending)
```

При дальнейшем развитии проекта количество индексов будет увеличиваться по мере появления новых сценариев поиска.
